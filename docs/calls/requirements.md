# Требования к видеосервису (Self‑Hosted Jitsi для психологов)

## Функциональные сценарии
- **Состав участников**: до 5 человек (типично: психолог‑хост + клиент; иногда супервизия).
- **Роли**:
  - **Хост (психолог)**: модераторские права (mute участников, remove/kick, lock room, end call).
  - **Гость (клиент)**: присоединение по ссылке, базовые действия (вкл/выкл камера/микрофон, чат, демонстрация экрана).
- **Создание комнаты**: с привязкой к записи (booking). Создаётся `session` со `roomId` и ограничением по времени.
- **Вход по персональным ссылкам**: backend генерирует короткоживущие JWT для хоста и гостя; без валидного JWT вход невозможен (даже зная `roomId`).
- **Продолжительность**: до 90 минут. За 15 минут до конца — предупреждение и видимый таймер. По истечении времени — всех выводим из комнаты, `session` переводим в завершённое состояние.
- **Pre‑join экран**: выбор камеры/микрофона, проверка звука/видео до входа.
- **Во время звонка**: включение/выключение аудио/видео, демонстрация экрана, чат, завершение звонка.
- **Записи**: в первой версии отсутствуют.

## Нефункциональные требования
- **Качество**: целимся в 720p с адаптацией качества (динамический битрейт). При деградации сети — приоритет стабильности.
- **Совместимость**: современные браузеры и устройства (desktop/mobile). Тестовый охват: Chrome, Firefox, Edge; Safari iOS; Chrome Android.
- **TURN обязателен**: поддержка TURN через UDP/3478 и TLS/5349. Даже локально — для проверки fallback‑сценариев.
- **Надёжность**: устойчивость к NAT/Firewall; корректный fallback на `relay`‑кандидата.
- **Производительность**: минимальные задержки для 1‑5 участников; приемлемое потребление CPU/RAM на JVB.
- **Наблюдаемость**: тех. логи (подключения, длительность, ошибки), без ПДн; дашборды позже (Prometheus/Grafana).

## Безопасность
- **WebRTC**: шифрование SRTP + DTLS.
- **Транспорт**: весь трафик по HTTPS; TURN с TLS/5349. Сервер за firewall, доступ по SSH‑ключам.
- **Аутентификация доступа в комнату**: JWT с TTL ≤ 90 мин, роли `host`/`guest`. Токены обязательны для входа.
- **Авторизация в Jitsi**: включён режим аутентификации (JWT); «гости» без токена отключены.
- **Хранение секретов**: в `.env`/Vault; ключи/секреты ротируются; ограничение прав доступа.
- **Удаление артефактов**: `join_tokens` истекают и удаляются; комнаты уничтожаются сразу после завершения.
- **Отсутствие записей**: исключает необходимость согласий на запись; соблюдаем приватность участников.

## Логирование и приватность
- **Логи**: только технические события (подключение/отключение, длительность, ошибки). 
- **ПДн**: не сохраняем; идентификаторы пользователей в логах — в виде хэшей (детерминированных на ключе).
- **Срок хранения**: ограниченный (например, 30–90 дней) с авто‑очисткой.

## Интеграция с текущим продуктом
- **UI «Записи»/Календарь**: кнопка «Создать комнату» у будущих встреч.
- **Рассылка приглашения**: /invite триггерит телеграм/email (или предоставляет ссылку для копирования).
- **После встречи**: обновляем статус записи, сохраняем ссылку на тех. лог.

## Ограничения и допущения
- До 5 участников, 720p — ориентир, но возможна адаптация качества.
- Jitsi self‑hosted (web/prosody/jicofo/jvb) + собственный coturn.
- На локальном стенде — self‑signed/mkcert; на VPS — валидные сертификаты и публичный домен.

## Состояния сессии
- **created** → **active** → **warning** (T‑15m) → **ended/expired**.
- Истечение времени переводит сессию в `expired`, повторный вход по токену невозможен.

## Критерии приёмки L1
- Описаны сценарии, нефункциональные требования и безопасность.
- Формализованы ограничения и минимальный набор логов без ПДн.
- Подготовлены архитектурные диаграммы и черновик локального сетапа.
